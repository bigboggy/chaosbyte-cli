# Chaosbyte — making the terminal come to life

**Status:** draft, awaiting Daniel's sign-off
**Date:** 2026-05-12
**Author:** Claude under Daniel's direction

## The mistake to avoid

The earlier framing of this design treated the engine as a *backdrop* — a separate animated layer behind the chat. That is wrong. Daniel was explicit: "having random words in the background is not the point. the feeling is to make the terminal come to life, not have cool art in the background."

The engine is not a layer. **The engine is a vocabulary of moves applied to the actual UI elements.** The chat itself. The game itself. The shared content. The countdown timer. The spotlight project name. The header chip. The input cursor. Each is a piece of content that owns its own layout and animation state. The engine is the verbs the renderer knows how to perform on those layouts.

This is the Cheng Lou *Pretext* architecture, applied to a terminal.

---

## The architecture (Pretext-style)

Pretext's breakthrough is the split between *layout* (expensive, do once) and *render* (cheap, do per frame). Every glyph is precisely addressable. Animations don't trigger reflow. That's what enables the wild creative work people have built on it — interactive text, ASCII video, masonry typography, per-glyph movement.

The same split, in our terminal:

```
internal/typo/
  layout.go       Layout — text wrapped into per-char (col, row) positions; immutable per content
  state.go        AnimationState — what's happening to a Layout right now (type progress, drop offset, tint, etc.)
  render.go       Render(layout, state) → cells; the only hot path, called per frame
  primitives.go   Type, Cascade, Wipe, Drop, Tint, Pulse, Scramble — state mutators
```

Each piece of content owns:
- One **Layout** (computed once on creation, cached). For a chat message: the wrapped text with each char's position. For a falling bar in bricks: the bar's character sequence with positions. For the spotlight title: the project name + position.
- One **AnimationState** (mutable, updated per tick). Holds: type-in progress, drop offset, tint color, scramble flag, fade alpha, cascade A→B state, etc.

Each frame, the Renderer walks every visible Layout, reads its AnimationState, emits cells. No grid backdrop. No composite-over-field. The actual UI content IS the engine's canvas.

This same renderer draws:
- Chat lines (Layouts that Type in on arrival)
- Game elements (Layouts that Drop, move, collide)
- The spotlight title (one Layout that Cascades on rotation)
- The header chip (a Layout that Cascades on screen switch)
- The bricks score (a Layout that Cascades on milestone)
- The opt-in countdown bar (a Layout that Scrambles in the last 5 seconds)
- ASCII video, if/when someone drops one in chat (a sequence of Layouts, one per frame)

**Single vocabulary. Single renderer. Per-glyph addressable.**

## Three principles

Same three as before — they survive the reframing.

1. **Quiet by default.** Most of the time, nothing is moving. Layouts at rest render static text. Animations fire on specific events and decay back. The room is still until something happens.

2. **Each move carries meaning.** A Type-in says "this just arrived." A Cascade says "this changed in place." A Drop says "this has weight, like a mod alert." A Scramble says "something's resolving, hold on." The vocabulary is a language.

3. **Decay is continuous, never abrupt.** The current `quiet()` cutoff snaps the field off. Wrong. AnimationState transitions are smooth interpolations, not state flips.

## The primitive vocabulary

Each primitive is a pure state mutator. Compose freely.

| Primitive | Effect on AnimationState | Mapped to which moments |
|---|---|---|
| **Type** | reveals chars left-to-right at ~50ms/char | New chat lines, code snippets, mod replies |
| **Cascade** | chars flap through long-ramp glyphs, settle on target | Spotlight title rotation, header chip on switch, score milestone |
| **Wipe** | progressive clear/reveal from one side | Submit (input clears right-to-left), screen transition, game start/end |
| **Drop** | chars descend from above with gravity acceleration | Mod alerts, falling chat bars in blitz |
| **Tint** | per-cell color override (RGB or theme color) | @mention line, whisper dim, build pass/fail flash |
| **Pulse** | brief intensity bump (per cell or per layout) | Typed character feedback, focused command palette item |
| **Scramble** | chars cycle randomly, settle back to target | Countdown last 5s, loading state, mod confusion |

These are what we ship. Anything else gets composed from these.

---

## The five categories (unchanged)

Each event belongs to exactly one. Category determines which primitive composition runs.

| Category | What it means | Primitive composition |
|---|---|---|
| **ARRIVAL** | someone/something showed up | Cascade or Type, ~3-5s |
| **ATTENTION** | this is for/about YOU specifically | Tint + Pulse + persistent foreground state |
| **SHIFT** | the room state changed | smooth palette interpolation, no glyph animation |
| **EVENT** | brief, decays | small Pulse on the row, ~1s |
| **TAKEOVER** | scene shifts entirely | Wipe out → scene → Wipe in |

## What "alive" looks like, surface by surface

**Chat scrollback.** Each `ChatMessage` gets a Layout on arrival. Default AnimationState applies Type primitive — chars reveal left-to-right at ~50ms/char (so a short line lands in <500ms, a long line in ~2s). Old messages (>5 min) get a slow Tint toward muted — reading history feels different from reading now. Mod posts get Drop primitive instead (gravity from top). `/me` lines get Scramble briefly. Whispers Type in at lower alpha.

**Input area.** Your textinput cursor leaves a Pulse-trail behind. Each keystroke applies Pulse to the cell just typed. Backspace dissolves the deleted char. Submit (enter) applies Wipe right-to-left to the input line, then the screen transitions.

**Member list.** New joins Type into their alphabetical position. Leaves Wipe out, gap closes. The typing nick Pulses at 1Hz while typing.

**Header chip.** Screen switch: chip Cascades from old title to new in place (split-flap board).

**Spotlight.** Title is one Layout. On rotation, Cascade from old project name to new. Highlight bullets Type in sequentially with stagger when card first renders. The "X:XX remaining" timer — each digit Drops down 1 cell when it changes.

**Bricks blitz.**
- Each falling chat-bar is a Layout with Drop state and accelerating velocity
- Paddle is a Layout, position state tracks last 3 positions for the streak
- Score is a Layout, Cascade primitive fires on each +50 milestone
- 30s timer is a Layout, Scramble in the last 5s, each digit
- Floor entries (missed bars) land with Tint flash
- End card title + score + lines Cascade simultaneously with stagger

**Shared content.** Repo URL drops Cascade the repo name above the message line, language tag Pulses once. Code snippets Type in line by line.

**Countdown timers.** Opt-in 15s bar fills with weight; at 5s the bar Tints warning; at 1s it's red and the digit Scrambles every frame.

**Slash commands.** The `/games` text dissolves chars right-to-left (Wipe + Tint dim) as the screen transitions. Palette filter items Wipe in/out as filter narrows.

## Games as text — the unlock

Once everything is Layout + AnimationState + per-glyph addressing, games and chat become the same medium. Same primitives. Same renderer.

**Bricks blitz** stops being a fullscreen TAKEOVER. It becomes a *region of the chat scrollback*. A user runs `/blitz` and a 30-second game starts as an 18-row block at the bottom of `#lobby`. Everyone subscribed sees it play out simultaneously in their own scrollback. When it ends, the end card stays in the scrollback as a record. You scroll back tomorrow and there's a frozen frame of yesterday's game with the score, like a screenshot embedded in the conversation.

**Future games drop into the same vocabulary:**
- Snake = a Layout where chars are body segments with positional state
- Asteroids = floating Layouts with rotation/movement state
- Tetris = a Layout for the locked grid + a Layout for the falling piece
- Pong = two paddle Layouts + a ball Layout

**Multiplayer is the broker.** Game state broadcasts via `room.Broker`. Each client renders the same Layout from the same shared state. Two people SSH'd in can play live; everyone else in the channel watches it happen.

**Replays are trivial.** State-over-time is serializable. Watch yesterday's blitz, frame by frame, in today's chat.

**Nobody has this in a terminal.** Discord can't do it. Slack can't do it. IRC obviously can't. Even sshtron is a single fullscreen game, not a game *in the chat*. This is the actual moat.

---

## ASCII video — same primitive, different content

A user drops an `.mp4` URL. The bot (or local handler) extracts a low-fps thumbnail sequence and renders each frame as a chars-as-pixels Layout. A short clip becomes a chat-row-tall sequence of Layouts the renderer plays at ~12-24fps in place. The video lives in the conversation like a GIF in Discord — except it's text, so it's searchable, copyable, and replayable forever.

Not P0. But the architecture *allows* it from day one, which is the point.

## Why this is the actual moat

Chaosbyte's signature claim is no longer "chat with cool effects." It is:

**The terminal where any participant can drop a game, a message, a video, or a moment into the conversation — and the room itself responds, in shared real-time, with the same vocabulary applied to every element.**

That's never been done. Discord chats with games are tabs you switch to. Slack with apps is a sidebar. IRC is dead-text. Chaosbyte is the first place where chat, game, video, animation, status all live in the SAME medium, sharing the SAME engine, viewable by EVERYONE simultaneously.

The Pretext architecture is what makes that single medium possible. Per-glyph addressable. Layout + state separated. One renderer. Per-character voice. The moat isn't the engine — it's that no other chat treats text as a fully-addressable substrate this way.

# Addendum 2026-05-12: ten critiques baked in

Signed off. The original plan was generic-terminal-effects with a slightly cleaner architecture. Ten changes make it Chaosbyte-specific.

## 1. Intention macros, not raw primitives

The public API is **macros**, not primitives. Screens and the mod speak in intentions. Macros decompose into primitive sequences with timing.

Initial macro set in `internal/typo/macros.go`:

| Macro | Composition | Used for |
|---|---|---|
| `Greet(nick)` | Type into chat + Cascade in member list + Pulse on header | new arrivals |
| `Amplify(layout)` | Tint accent + Pulse + slight scale-up | something the room should attend to |
| `Mourn(layout)` | slow Wipe + Echo afterimage | departures, ended spotlights |
| `Storm()` | simultaneous Drop + red Tint across multiple Layouts | build broke, deploy failed, serious moments |
| `Echo(target, source)` | render a ghost of `source` above `target` | replies, "as you said" callbacks |
| `Quiet(layout)` | dim Tint + slow alpha fade | old messages decaying out |
| `Settle(layout)` | Scramble then resolve, ~1s | mod-led arrivals, things that "decide what they are" |

Primitives become an implementation detail. Screens never call Type or Cascade directly — they call `Greet` or `Amplify`. This is the durable surface.

## 2. The mod is the director, not wallpaper

Rules-v0 currently posts canned lines. That's wrong for what could be the moat. The mod is the room's choreographer:
- Decides which message to `Amplify`
- Calls `Greet` for arrivals (instead of the screen calling it directly)
- Queues spotlights with intent (this person dropped a repo → queue them next, mention why)
- Calls `Storm` when a build breaks
- Calls `Quiet` when the room's been frantic and needs a beat

Architecture: `internal/mod` grows a `Director` interface.
- v0: heuristics on broker state pick macros (this week)
- v1: LLM-backed (Claude API via tool-use); system prompt knows the macro vocabulary + room context. The mod literally writes the moments.

## 3. Per-user voice

Every user has a `Style` that modulates how their content arrives:

```go
type UserStyle struct {
    Entrance Primitive    // Type / Drop / Cascade / Scramble
    Voice    EasingCurve  // slow / quick / weighted
    Drift    Direction    // leftward / centered / rightward bias
    Tint     lipgloss.Color  // optional signature accent
}
```

`@boggy`: slow Type with leftward drift. `@yamlhater`: fast Drop. `@ai_grifter`: Scramble before settle. Default for new users: fast Type, nick-color tint.

Style is part of broker user state. Pulled when rendering each message. Mod can assign signature styles to characterful regulars.

Lives in `internal/typo/style.go`. This is the actual Chaosbyte differentiator — personality lives in motion.

## 4. Games as conversational artifacts (not just inline regions)

The "inline games-in-chat" framing was right but incomplete. Going further:

- A finished game writes a `GameRecord` to chat as a permanent `ui.ChatGameRecord` message kind
- The record is a chat post — scroll back, link-share, `/replay <id>` it months later
- Replies to a game record start a re-match thread
- `/fork <id>` clones the playfield/rules and posts your variant as a new game
- `#blitz-tournament` channels emerge naturally — they're just channels of game records

`internal/games/record.go` defines `GameRecord` serialization (compact JSON of state + replay timeline). Broker treats them like any chat. The chat scrollback becomes the canonical record of every game played.

## 5. Reactions animate the source, no emoji tails

Reactions don't append `🔥 👍 ❤️` after a message. They visually modify the source:

| Reaction | Effect on source message Layout |
|---|---|
| 🔥 | Tint warm + Pulse for 3s |
| ❤️ | soft pink halo, persists 5s |
| 👎 | message Drops slightly (slumps) |
| 🎯 | brief Cascade on the body |
| ☕ | message Tints muted (chill mood) |

Multiple reactions stack — a `🔥` followed by `❤️` warms then haloes. `internal/typo/reactions.go` maps reaction kind → macro applied to the source.

## 6. Threading as nested Layouts

Replies are Layouts visually nested under their parent:
- Indented ~4 cols
- Connection line drawn with box-drawing chars (`└─`) from parent's left edge to child
- Parent stays visible above children even when older
- Long threads (>3 replies) compress to a `+N more` Layout that expands on click/enter

Chat scrollback becomes a tree, not a list. `internal/typo/thread.go` owns the depth + connection drawing.

## 7. Latency: optimistic local + broker reconcile

For multiplayer scenes (games-in-chat, reactions, etc.):
- Local input mutates local Layout state immediately
- State broadcasts to broker async
- Other clients render the broadcasted state
- Local mispredict (game disagrees with broker authority) reconciles with a brief Wipe → re-render

`internal/room/sync.go` handles optimistic state management. Without this the games-in-chat work will feel laggy.

## 8. Pretext analogy: honest about what we're taking

We're taking Pretext's **programming model** (layout immutable, state mutable, per-glyph addressable, render is a hot read), not its **performance win** (we don't have a DOM-reflow problem in monospace terminal). The abstraction is the value; the speed claim doesn't translate.

## 9. SHIFT category: commit, don't half-build

SHIFT was "atmospheric backdrop change" — backdrop thinking. Reframed: SHIFT means **every visible Layout's AnimationState gets a coordinated Tint pulled toward the new mood, with stagger**. Not a global palette flag — a re-tinting of all room content simultaneously, visible on every message, game, chip.

If we can't commit to that, we drop SHIFT as a category entirely and roll those events under EVENT with Tint.

## 10. Mod-cued tutorial (discoverability)

First-ever join (no prior nick in broker) → mod runs a 5-beat intro:

1. `Greet` you (ARRIVAL example)
2. Drop a mod prompt explaining where to find help (Type example)
3. Invite you to @mention yourself (ATTENTION example)
4. Show one game record from the channel (artifact example)
5. Drop a quiet moment so you see what default state looks like (SHIFT example)

Tutorial is itself a conversation, not a manual page. Lives in `internal/mod/tutorial.go`.
